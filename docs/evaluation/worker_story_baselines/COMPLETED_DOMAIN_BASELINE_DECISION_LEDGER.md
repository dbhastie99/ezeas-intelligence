# Completed Domain Baseline Decision Ledger

Worker Story Evaluation Baselines v0.1 records the current Minerva product-domain maturity inventory and decides whether a follow-up checked-in baseline-capture slice is justified.

This ledger is diagnostic/control only. It does not mutate corpus, does not change routing, does not change answer generation, does not call live LLM, does not ingest operational JSON, does not connect Code Evidence, does not prove runtime platform truth, and does not create v0.5 slices automatically.

## Scope And Decision Rules

The inventory was compiled from repo evidence: domain retrieval plans in `app/services/domain_retrieval_plan_service.py`, rich-answer benchmark manifests in `samples/eval/`, corpus coverage services/scripts in `app/services/` and `scripts/`, answer gap report services/scripts in `app/services/` and `scripts/`, evaluation runbooks in `docs/`, README pointers, tests, and the absence of checked-in baseline/evaluation-result artefact folders.

DecisionStatus values used in this ledger:

- `BASELINE_REQUIRED`
- `BASELINE_ALREADY_EXISTS`
- `NO_BASELINE_NEEDED`
- `RUNBOOK_OUTSTANDING`
- `NEEDS_REVIEW`

Decision rules:

- `BASELINE_REQUIRED`: evaluation runbook exists, benchmark/diagnostic/gap-report commands or equivalent evaluation outputs exist, no checked-in baseline artefact exists, and future before/after comparison is useful.
- `BASELINE_ALREADY_EXISTS`: checked-in baseline/evaluation result artefacts already exist.
- `NO_BASELINE_NEEDED`: no meaningful comparative evaluation output exists, or the stream does not need a separate baseline beyond v0.4 runbook and tests.
- `RUNBOOK_OUTSTANDING`: v0.1-v0.3-style artefacts exist but the v0.4 evaluation runbook foundation or required domain-specific diagnostic/gap tooling remains outstanding.
- `NEEDS_REVIEW`: repo evidence is ambiguous.
