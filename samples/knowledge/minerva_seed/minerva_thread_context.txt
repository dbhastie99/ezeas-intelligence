Platform Update

We are commencing implementation of the Ezeas Intelligence Layer, now named Minerva.

Minerva is a focused platform capability, not a generic chatbot. It is the Ezeas knowledge and evidence intelligence layer that will allow users to ask natural-language questions over:

Developer Logs
Hardening Logs
Platform Doctrine
client requirements and responses
curated project chat history
PayRun JSON artefacts
Worker Story JSON
Movement Review evidence
Payroll Bases & Totals evidence
exception payloads
correction audit evidence
finalisation readiness evidence
reconciliation outputs

Minerva’s purpose is to make Ezeas’ deterministic evidence estate searchable, explainable, comparable and operationally useful.

The deterministic Ezeas platform remains the source of payroll truth. Minerva must not calculate payroll, determine entitlements, interpret awards at runtime, approve exceptions, suppress warnings, override payroll outcomes, mutate configuration or finalise PayRuns. Minerva reads, explains, summarises, clusters, compares and guides.

The agreed architecture is a separate Python/FastAPI service in a new repo:

ezeas-intelligence

with a separate SQL Server database:

ezeas-intelligence-db

The operational Ezeas database remains authoritative for payroll, leave, award, workforce and reconciliation truth. Minerva may reference operational IDs such as AccountId, PayRunId, ContactId, ProcessPeriodId and ArtifactId, but it must not become operational payroll truth.

Initial implementation should use SQL Server, document chunks, source metadata, simple keyword/full-text retrieval and audit logging. Do not introduce Postgres, pgvector, complex agent frameworks or live payroll JSON in the first slice.

Thread Context

The prior thread established the full conceptual and governance model for Minerva.

Key decisions already made:

Name decision

The Ezeas Intelligence Layer is now called Minerva.

Unified architecture decision

Minerva supports both:

Platform Knowledge Assistant
Operational Evidence Intelligence Assistant

These are not separate products. They share one secure ingestion, indexing, retrieval, redaction, source-authority, answer-generation and audit architecture.

LLM boundary decision

Minerva is explanatory and advisory only. It does not calculate payroll or mutate payroll truth.

Separate database decision

Minerva uses its own SQL Server database, ezeas-intelligence-db.

SQL Server first decision

Use MS SQL Server for the local prototype. Vector search can come later.

No raw JSON by default decision

Operational JSON evidence must be registered, hashed, classified and extracted into facts/summaries before LLM use. Raw payroll JSON should not be sent wholesale to an LLM by default.

RBAC-before-LLM decision

Tenant filtering and user permissions must be applied before evidence reaches the LLM.

Source authority decision

Platform Doctrine, Hardening Logs and Developer Logs outrank raw chat history. Chat history is supporting material, not authoritative if it conflicts with formal doctrine/logs.

No autonomous AI action decision

Minerva may recommend review or explain evidence. Suppression, override, repair, finalisation and configuration changes remain deterministic workflows requiring governed user action.

Logging decision

Minerva work remains in the broader Workforce Platform Developer Log / Hardening Log / Platform Doctrine strategy, even though the repo is separate. Separate repo is an engineering boundary; shared logs and doctrine are the platform memory boundary.

Developer Log structure decision

All future Developer Logs must include:

User Guide / Rationale and Operating Model

The Still To Do / Do Not Lose section must now act as a continuity audit checkpoint, carrying forward current-thread items, active platform commitments and deferred no-status-change commitments.
